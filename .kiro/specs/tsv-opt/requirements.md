# Requirements: tsv-opt

`H2-main_TSV_Opt` を、特定の解析シナリオに依存しない汎用的な熱解析フレームワークとして再構築し、TSV最適化のための高度な機能を安定して提供する。

## 1. 現状の課題 (Current Gaps)

- **ハードコードの散在**: 13レイヤーの定義や特定のGDSIIファイル名が `modelA.jl` や `heat3ds.jl` に直接記述されている。
- **エンジンの密結合**: ソルバーロジックが物理モデル（材料や境界条件）と密接に結合しており、他のモデルへの差し替えが困難。
- **設定の不完全さ**: JSON 設定ファイルはあるが、プログラム内の全てのパラメータをカバーできていない。
- **テストの欠如**: 主要な計算パスや並列処理の正当性を検証する自動テストが不足している。

## 2. 機能要件 (Functional Requirements)

### 2.1 構成管理の刷新
- **フル・コンフィギュレーション**: 全てのジオメトリ、材料、境界条件、ソルバーパラメータを `config.json` から制御可能にする。
- **モデル・インジェクション**: `modelA.jl` のような具体的なモデルを、コアエンジンからプラグインとして扱えるように分離する。

### 2.2 解析機能の強化
- **TSV 最適化指標の自動化**: `compute_metrics` を拡張し、設計者に有用な熱抵抗やホットスポット情報を自動出力する。
- **非一様格子の汎用化**: Z軸だけでなく、X, Y軸についても設定ファイルからグリッド密度を制御可能にする。

### 2.3 データ連携
- **GDSII パースの安定化**: 現在の `load_gds_geometry!` を見直し、複数のレイヤーや複雑なパターンにも柔軟に対応可能にする。

## 3. 非機能要件 (Non-Functional Requirements)

- **パフォーマンス**: Julia のマルチスレッド機能を最大限に活かし、現状と同等以上の計算速度を維持する。
- **保守性**: `src/` 配下のコアロジックは、特定の実験ディレクトリから独立して利用可能にする。
- **型の安全性**: Julia の型システムを適切に利用し、ランタイムエラーの早期発見と最適化を行う。

## 4. 成功の定義 (Definition of Success)

- `H2-main_TSV_Opt` が、コアエンジンの修正なしに、異なる `config.json` とモデル定義のみで動作すること。
- TSV 配置の変更が、シミュレーション結果に即座に反映され、評価指標が自動で算出されること。
- 新しい構成において、`Heat3D-original` と比較して計算精度が維持されていることが検証されること。
