# Requirements: tsv-opt (Model Expansion)

`Heat3D-original` をベースとし、シリコンチップの形状（GDSII）や TSV 配置（JSON）を外部ファイルから柔軟に定義・解析できる汎用的な熱解析フレームワークを構築する。

## 1. 背景と目的 (Background & Goals)

- **ベースの固定**: 学習・検証済みの `Heat3D-original` をメインプロジェクトとして採用し、その信頼性を維持する。
- **モデルの外部定義**: 形状や構造をコードの修正なしに変更可能にすることで、多様なモデルの解析を容易にする。
- **設計の整理**: `H2-main_TSV_Opt` で実現されている機能を整理し、拡張性と保守性の高いコードベースを再構築する。

## 2. 機能要件 (Functional Requirements)

### 2.1 形状・構造の外部定義
- **シリコンチップ (GDSII)**: GDSII ファイルを読み込み、チップ内の配線パターンや領域をグリッド上の物性値（ID）として反映する。
- **TSV 配置 (JSON)**: TSV の本数、中心座標 (x, y)、および太さ（直径/半径）を JSON ファイルから定義可能にする。
- **モデル・インジェクター**: `Heat3D-original` の `setMaterial!` や `setMask!` に相当する部分を、外部定義に基づいた動的な生成プロセスに置き換える。

### 2.2 構成管理 (Future-proofing)
- **JSON 駆動への道筋**: 現時点では実装不要だが、将来的に「各層の厚さ」「物性値」「層数」なども JSON から与えられるような拡張性を考慮したデータ構造とする。

### 2.3 ソルバーの維持
- `Heat3D-original` が持つ反復解法（SOR, RB-SOR等）をそのまま利用し、モデルの複雑化に伴う計算精度の検証を可能にする。

## 3. 非機能要件 (Non-Functional Requirements)

- **コードの可読性**: ライブコーディングによる密結合を避け、物理モデル定義とソルバー計算を明確に分離する。
- **参照制限**: `H2-main_TSV_Opt` は実装のヒントとして参照するのみとし、コードの直接的なコピーは行わない。

## 4. 成功の定義 (Definition of Success)

- 異なる GDSII ファイルや TSV 設定（JSON）を読み込むだけで、シミュレーションが正常に実行され、熱分布が正しく計算されること。
- `Heat3D-original` のディレクトリ構造を壊さず、追加機能として綺麗に統合されていること。
